# Lessons Learned

## Architecture

- Код и миграции должны выигрывать у обзорных документов при любом конфликте.
- Общий layout нельзя перегружать последовательными auth и tenant lookup для каждой страницы.

## Supabase и RLS

- RLS по `club_id` не заменяет гранулярную проверку прав в Server Actions.
- Service role требует ручного tenant scope даже при корректных RLS policies.
- Массовые security-правки необходимо проверять по каждой функции: автоматическая замена может оставить действие без проверки.

## Performance и Vercel

- Перед глубокой оптимизацией запросов нужно проверить физическое расположение сервисов и сетевую задержку между ними.
- Сначала измеряются round trips и критический путь, затем добавляются RPC, параллелизм или индексы.

## UX

- Desktop toolbar нельзя просто сжимать на мобильном: заголовок, поиск и фильтры должны переходить на отдельные строки.
- Drawer, overlay и sidebar проверяются на реальном мобильном viewport, а не только по CSS-классам.

## AI workflow

- В handoff должны попадать факты, решения, риски и следующий шаг, но не полный transcript.
- Автоматизация документации может менять только явно размеченные секции.

## Testing и Deployments

- Успешный build не доказывает корректность RLS, webhook или пользовательского сценария.
- Deploy нельзя отмечать успешным только по наличию commit; нужна фактическая проверка Vercel/production.
