# Хранить нераспознанные поля импорта в clients.import_data

## Контекст

FitCRM принимает выгрузки из CRM с различными наборами колонок. Каноническая модель клиента должна оставаться пригодной для фильтров, отчётов и интеграций, но исходные данные нельзя молча терять.

## Проблема

Создание отдельной SQL-колонки под каждое поле внешней CRM не масштабируется. Добавление неизвестных значений в `notes` смешивает машинный импорт с заметками сотрудников и лишает данные структуры.

## Рассмотренные варианты

1. Отбрасывать неизвестные поля.
2. Добавлять их текстом в `clients.notes`.
3. Создать EAV-таблицу custom fields.
4. Хранить provenance последнего импорта и неизвестные поля в `clients.import_data jsonb`.

## Решение

Использовать канонические SQL-поля для известных данных и `clients.import_data` для `sourceFile`, `sourceRow`, `importedAt` и `extraFields`. При повторном update дополнительные поля объединяются по названию, новое значение побеждает. Карточка клиента показывает их отдельным read-only блоком.

## Причины

- Нет потери данных при импорте из нового источника.
- Основная схема и отчёты не зависят от произвольных полей внешних систем.
- Заметки сотрудников не загрязняются техническими данными.
- JSONB достаточно для редкого чтения внутри одной карточки и не требует отдельного join.

## Последствия

Фильтры и аналитика работают только по каноническим полям. Для массовой фильтрации конкретного дополнительного поля его нужно будет повысить до канонического столбца или вынести в отдельную модель custom fields.

## Риски

Один клиент хранит только последнее происхождение импорта, хотя дополнительные значения объединяются. Размер ограничивается сервером: до 100 полей, имя до 120 символов, значение до 2000 символов.

## Условия пересмотра

Появление пользовательского конструктора custom fields, необходимость истории нескольких импортов или массовой аналитики по внешним полям.
