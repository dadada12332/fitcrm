# Клиентские обращения как отдельный channel-agnostic inbox

## Контекст

В FitCRM уже есть `support_*` для общения клуба с командой платформы. Telegram Mini App потребовал другой поток: клиент конкретного клуба обращается к сотрудникам этого клуба, а ответ возвращается через бот клуба. В будущем тот же рабочий стол должен принимать Instagram и web-каналы.

## Проблема

Переиспользование платформенных тикетов смешало бы разные аудитории, permissions, SLA и tenant-модель. Отдельный Telegram-only экран, напротив, пришлось бы дублировать для каждого нового клиентского канала.

## Рассмотренные варианты

1. Расширить `support_*` платформы клиентскими сообщениями.
2. Создать Telegram-only таблицы и экран.
3. Создать отдельный tenant-scoped inbox с общими conversations/messages и полем channel.

## Решение

Выбран вариант 3. `client_conversations` и связанные таблицы принадлежат клубу и клиенту CRM; Telegram является первым значением `channel`. Платформенная поддержка остаётся независимой. Каноническая идентичность клиента — внутренний `clients.id`, полученный из подтверждённой Telegram-связи.

Authenticated получает только RLS-scoped read. Все записи CRM выполняются permission-checked Server Actions, а записи Mini App — service-role API после HMAC-проверки `initData` и ручного `club_id/client_id` scope. Доставка ответов отделена от сохранения и имеет idempotency, статусы и cron retry.

## Причины

- Один понятный inbox для сотрудников и будущих каналов.
- Нельзя смешать обращение клиента с запросом клуба в FitCRM support.
- Стабильная идентичность не зависит от Telegram username или отображаемого имени.
- Сохранение сообщения не теряется при временной ошибке Telegram.

## Последствия

- Добавлен новый permission module `inbox` и отдельный unread counter.
- Instagram messaging должен подключаться к этой модели через `channel='instagram'`.
- Для realtime используется изменение conversation после вставки message.
- Ответы клиенту зависят от подключённого клубного бота; сбои видны сотруднику и повторяются cron.

## Риски

- Service-role пути требуют обязательного ручного tenant scope.
- Без подключённого бота ответ сохраняется, но не доставляется; UI обязан показывать failed state.
- Метрики SLA и вложения пока не входят в первую версию.

## Условия пересмотра

Пересмотреть модель, если Instagram API требует отдельной thread identity, появляется омниканальное объединение контактов или объём сообщений требует очереди вместо cron batch retry.
