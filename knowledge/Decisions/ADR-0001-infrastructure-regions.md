# Vercel и Supabase размещаются в близких регионах

## Контекст и проблема

FitCRM выполняет Server Actions и Server Components в Vercel, а данные получает из Supabase Postgres. Каждый последовательный auth/RPC/query добавляет сетевой round trip. При удалённых регионах даже быстрый SQL создаёт заметную задержку во всех разделах CRM.

Подтверждённый факт: `vercel.json` задаёт `syd1`. Старый `FITCRM_ARCHITECTURE.md` указывает `iad1`, поэтому он не является источником текущей конфигурации. Фактический регион Supabase не подтверждён.

Наблюдавшиеся симптомы: медленная загрузка таблиц и переходов, длинная последовательная цепочка auth/club запросов. Анализ `PERF_REPORT.md` обнаружил лишние round trips; после их сокращения медианный TTFB в зафиксированном замере снизился примерно с 157 до 127 мс. Эти данные доказывают стоимость round trips, но сами по себе не доказывают межрегиональное размещение.

## Рассмотренные варианты

1. Оставить регионы как есть и оптимизировать только код.
2. Разместить Vercel compute рядом с текущим Supabase.
3. Перенести Supabase ближе к Vercel с миграционным окном.

## Решение

Vercel compute и Supabase Postgres должны находиться в одном или географически близких регионах. Перед переносом необходимо подтвердить оба региона и собрать baseline. Конкретное направление переноса выбирается после оценки данных, downtime и rollback.

## Причины

Это уменьшает базовую latency каждого server-side обращения и защищает все страницы, включая те, где round trips уже оптимизированы.

## Последствия и риски

- Регион становится частью архитектурной конфигурации и launch checklist.
- Перенос БД может требовать downtime, повторной настройки Auth/webhooks и проверки storage.
- Нельзя выполнять перенос без backup, restore rehearsal и rollback plan.

## Метрики

- p50/p95 server-side DB round-trip.
- p50/p95 TTFB авторизованных страниц.
- Cold start отдельно от warm request.
- Число последовательных auth/RPC/query на критическом пути.
- Error rate и время callback Payme/Click.

## Предотвращение повторения

Environment Matrix хранит подтверждённые регионы; performance audit начинается с topology check. После изменения инфраструктуры метрики снимаются повторно.

## Условия пересмотра

Появление edge read replicas, смена основного рынка или доказательство, что другой регион даёт лучшие end-to-end показатели без риска данным.
