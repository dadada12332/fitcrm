# FitCRM AI Workflow

Этот файл — единый источник правил обновления проектной памяти для Codex, Claude и других AI-агентов.

## Перед работой

1. Прочитать `AGENTS.md`, `knowledge/10 AI Handoff.md` и документы, указанные в handoff.
2. Проверить `git status`; не перезаписывать чужие незакоммиченные изменения.
3. Проверять пути, имена функций и состояние реализации непосредственно в коде.
4. Создать или обновить `knowledge/03 Current Task.md` для значимой задачи.

## Источники истины

- Код — фактическая реализация.
- `supabase/migrations/` — схема и история базы данных.
- Git — история изменений.
- Figma и `DESIGN_SYSTEM.md` — UI и дизайн-система.
- `knowledge/02 Current State.md` — оперативное состояние.
- `knowledge/05 Kanban.md` — состояние задач.
- ADR — причины долгосрочных решений.
- `knowledge/06 Changelog.md` — пользовательски значимые изменения.

При конфликте документации с кодом не угадывать: проверить код, исправить документ и отметить это в Daily.

## После значимой задачи

1. Проверить `git diff`.
2. Обновить или закрыть задачу и Kanban.
3. Запустить `npm run docs:daily`.
4. Обновить Changelog только для пользовательски значимых изменений.
5. Создать или обновить ADR, если принято долгосрочное решение.
6. Обновить Known Issues и Current State, если их состояние изменилось.
7. Запустить `npm run docs:handoff`.
8. Запустить `npm run docs:validate`.
9. Записать фактически выполненные проверки. Не объявлять tests, build или deploy успешными без запуска.

Для типового завершения используется `npm run docs:sync`; команда обновляет только явно размеченные автоматические секции.

## Защита памяти

- Не сохранять chat transcript, внутренние рассуждения AI и полный git diff.
- Не сохранять секреты, токены, пароли, credentials и персональные данные клиентов.
- Не создавать отдельный task-файл для каждой мелкой правки.
- Не записывать предположения как факты; использовать `Не проверено` или `Нет доступных данных`.
- Не удалять старую документацию без анализа и явного решения о миграции.
- Если информация не меняет состояние, решение, задачу, риск или обобщаемый вывод, её не нужно сохранять.

## Git

- Документация валидируется перед commit через `.githooks/pre-commit` после однократного `npm run docs:hooks`.
- Hook ничего не генерирует и не изменяет, а только запускает проверку.
- Commit и push выполняются только в рамках явных договорённостей с владельцем.
