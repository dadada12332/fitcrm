## 1. Общая информация

**Статус раздела: ✅ Готово**

### Описание продукта

**FitCRM** — облачная (SaaS) CRM-система для управления фитнес-клубами и спортивными студиями. Продукт ориентирован на рынок **Узбекистана** (валюта UZS «сум», язык интерфейса — русский, таймзона Asia/Tashkent).

### Цель

Дать владельцу фитнес-клуба единый инструмент для управления клиентами, абонементами, посещениями, финансами, персоналом, складом и коммуникациями (Telegram), а оператору платформы — инструмент управления всеми клубами-арендаторами (Platform Admin).

### Для кого

| Аудитория | Что получает |
|---|---|
| Владелец клуба | Полный контроль: клиенты, деньги, персонал, аналитика |
| Администратор / менеджер | Ежедневные операции: продажи абонементов, приём оплат, check-in |
| Тренер | Свои клиенты, расписание, отметки посещений |
| Бухгалтер | Финансы, отчёты, зарплаты |
| Кассир | Продажи и посещения на ресепшене |
| Оператор платформы (SaaS) | Управление всеми клубами, подписками, здоровьем аккаунтов |
| Клиент клуба | Telegram-бот: свой абонемент, QR для входа |

### Какие задачи решает

- Учёт клиентов и их абонементов (продажа, продление, заморозка).
- Контроль посещений (ручной / QR / Telegram).
- Приём и учёт оплат, отчётность по выручке.
- Управление персоналом и правами доступа.
- Складской учёт товаров.
- Аналитика клуба (9 вкладок отчётов + AI-инсайты на эвристиках).
- Коммуникация с клиентами через Telegram (бот + рассылки).
- Импорт существующей базы клиентов из Excel/CSV.

### Основные преимущества

- **Server-first архитектура** — быстрые страницы, тяжёлые вычисления на сервере (SQL-агрегация).
- **Мультитенантность из коробки** — строгая изоляция клубов через RLS.
- **Гибкий RBAC** — 6 системных ролей + кастомные роли на клуб.
- **Platform Admin** — полноценная админка SaaS с health-score и support-режимом.
- **Локализация под рынок** — UZS, русский, Telegram как основной канал.

### Архитектура SaaS

Одно приложение Next.js, одна БД Postgres (Supabase), изоляция по `club_id` через RLS. Две плоскости: **CRM** (для клубов) и **Platform Admin** (для оператора SaaS, домен `admin.fitcrm.uz`). Подробности — [FITCRM_ARCHITECTURE.md](./FITCRM_ARCHITECTURE.md).

## 3. Архитектура проекта

**Статус раздела: ✅ Готово** · Детально — [FITCRM_ARCHITECTURE.md](./FITCRM_ARCHITECTURE.md)

### Кратко

- **Frontend + Backend:** Next.js 16 (App Router) — серверные компоненты, Server Actions, RPC.
- **Database / Auth / Storage:** Supabase (Postgres + RLS).
- **Hosting:** Vercel.
- **Изоляция арендаторов:** RLS по `club_id` + `user_club_ids()`.
- **Service-role:** обход RLS для платформы и после проверки прав.
- **Middleware:** рефреш сессии + rewrite домена `admin.fitcrm.uz → /platform` + защита роутов.
- **Интеграции:** Telegram (grammy), Storage; платёжные шлюзы ❌ не подключены.

### ASCII-схема потока

```
Landing (marketing)
        │
        ▼
   /login  /register  ← Supabase Auth (email, Google, phone-OTP через Telegram)
        │
        ▼
  app.fitcrm.uz  →  (app) CRM  ──── Server Actions / RPC / RLS ────┐
        │                                                          │
        ▼                                                          ▼
   Platform (admin.fitcrm.uz → /platform)  ── service-role ──►  Supabase
        │                                                    Postgres · Auth · Storage
        └────────────────────────────────────────────────────────┘
                                                                   │
                                          ┌────────────────────────┴───────────┐
                                          ▼                                     ▼
                                    Telegram Bot                          Vercel Cron
                                  (webhook / OTP / QR)             (daily-report 04:00, broadcasts 03:00)

  Payments (Click/Payme/Uzum) ── ❌ не подключены (только enum-лейблы)
```

## 6. Структура базы данных

**Статус раздела: ✅ Готово (25 таблиц)**

Схема в `supabase/migrations/0001_init.sql` (18 таблиц ядра) + расширения в 0010/0012/0013/0023/0027.

### 6.1 Таблицы ядра CRM

| Таблица | Назначение | Ключевые поля | RLS |
|---|---|---|---|
| `users` | Профиль поверх `auth.users` | `id`, `email`, `full_name`, `platform_role` (0023) | только своя строка |
| `clubs` | Клуб (арендатор) | `id`, `name`, `city`, `owner_id`, `settings` jsonb, `plan`, `trial_expires_at`, `plan_expires_at`, `tg_token`, `status`/`health_score`/`admin_notes`/`suspended_at` (0023) | SELECT членам, UPDATE владельцу |
| `staff` | Членство user→club + роль | `user_id`, `club_id`, `role` (text), `salary`, `is_active`, `settings` (0009) | SELECT членам клуба |
| `club_roles` | Кастомные роли клуба | `club_id`, `key`, `name`, `permissions` jsonb, `is_system` | SELECT членам, ALL владельцу |
| `clients` | Клиенты | `club_id`, `full_name`, `phone`, `telegram_id`, `qr_token`, `photo_url`, `tags[]`, `notes`, `birth_date`/`gender`/`source` (0007/0022), `balance` (0019), `debt`/`trainer_name` (0022) | club-scoped |
| `memberships` | Типы абонементов | `club_id`, `name`, `price`, `duration_days`, `visits_limit`, `freeze_days_allowed`, `is_active`, `archived` (0006) | club-scoped |
| `subscriptions` | Абонементы клиентов | `club_id`, `client_id`, `membership_id`, `starts_at`, `expires_at`, `visits_total`, `visits_used`, `status`, `frozen_at`, `freeze_days_*` | club-scoped |
| `visits` | Посещения | `club_id`, `client_id`, `subscription_id`, `checked_in_at`, `checked_in_by`, `method` (manual/qr/telegram) | club-scoped |
| `payments` | Платежи | `club_id`, `client_id`, `subscription_id`, `amount`, `currency`, `provider`, `tx_id`, `idempotency_key`, `status`, `paid_at` | club-scoped |
| `rooms` | Залы | `club_id`, `name`, `capacity`, `is_active` | club-scoped |
| `schedules` | Шаблоны расписания | `club_id`, `staff_id`, `room_id`, `title`, `day_of_week`, `start_time`, `end_time` | club-scoped |
| `classes` | Конкретные занятия | `club_id`, `schedule_id`, `staff_id`, `room_id`, `date`, `start_time`, `seats_total/booked`, `status` | club-scoped |
| `class_bookings` | Записи на занятия | `club_id`, `class_id`, `client_id`, `subscription_id`, `status` | club-scoped |
| `products` | Товары склада | `club_id`, `name`, `category`, `unit`, `sell_price`, `buy_price`, `sku`, `is_active` | club-scoped |
| `inventory` | Остатки | `club_id`, `product_id`, `quantity`, `min_quantity` | club-scoped |
| `stock_movements` | Движения склада | `club_id`, `product_id`, `type`, `qty`, `unit_price`, `client_id`, `payment_id`, `note` | club-scoped |
| `notification_templates` | Шаблоны уведомлений | `club_id`, `name`, `channel`, `body` | club-scoped |

- **Осталось:** —
- **Таблицы:** clients, subscriptions, visits, payments, staff.
- **RPC:** `get_dashboard_stats`, `get_layout_context`.

### 7.2 Клиенты — ✅ Готово

- **Описание:** список клиентов с серверной пагинацией/поиском/фильтром/сортировкой.
- **Возможности:** статус (активный/истёк/заморожен) вычисляется из подписок в SQL; фильтр-дровер; создание клиента; экспорт CSV; импорт из Excel/CSV.
- **Реализовано:** список, счётчики, CRUD, импорт, экспорт.
- **Осталось:** —
- **Таблицы:** clients, subscriptions, memberships.
- **RPC:** `clients_page`, `clients_stats`.

### 7.3 Карточка клиента — ✅ Готово

- **Описание:** профиль клиента (`clients/[id]`) с вкладками.
- **Возможности:** абонементы, посещения, платежи, заморозка/продление, заметки, теги, баланс/долг, тренер.
- **Таблицы:** clients, subscriptions, visits, payments.
- **Логика:** `lib/client-profile.ts`.

### 7.4 Посещения — ✅ Готово

- **Описание:** учёт check-in.
- **Возможности:** быстрый check-in, KPI, таблица с историей, продление; методы `manual/qr/telegram`; серверная пагинация.
- **Таблицы:** visits, subscriptions, clients.
- **RPC:** `visits_page`, `get_visits_kpi`.

### 7.5 Оплаты — ✅ Готово (учёт вручную)

- **Описание:** учёт платежей.
- **Возможности:** KPI, фильтры, таблица, модалка/дровер новой оплаты, пагинация; провайдер (cash/click/payme/uzum — **лейбл**), статус, idempotency_key.
- **Осталось:** ❌ реальный приём онлайн-оплат (шлюзы не подключены).
- **Таблицы:** payments, subscriptions, clients.
- **RPC:** `payments_page`, `get_payments_kpi`.

### 7.6 Абонементы (memberships) — ✅ Готово

- **Описание:** каталог тарифов клуба.
- **Возможности:** карточки-градиенты, создание/редактирование, архив, продажа (создание subscription); цена, длительность, лимит посещений, дни заморозки.
- **Таблицы:** memberships, subscriptions.

### 7.7 Расписание — ✅ Готово (~90%)

- **Описание:** управление загрузкой клуба.
- **Возможности:** шаблоны (schedules), тренеры, залы.
- **Осталось:** 🚧 полноценный поток занятий/бронирований (classes/class_bookings) — схема есть, UI частичен.
- **Таблицы:** schedules, classes, class_bookings, rooms, staff.

| **Финансы** | `reports_finance` | Выручка, платежей, ср. чек, выручка/день, дельта vs прошлый период; по провайдерам; по дням | area (выручка), donut (провайдеры) | период |
| **Продажи** | `reports_sales` | Продано, выручка, число тарифов; разбивка по тарифам (кол-во/выручка/доля) | bar (топ тарифы) | период |

- Финансовый `prevRevenue` и долги считаются по **всей** истории (RPC), тогда как старый браузерный код видел только `created_at ≥ год` — это исправление «слепого пятна года».
- Heatmap/пик/тихий день считаются в TZ `Asia/Tashkent`; посуточные графики — по UTC-датам (сохранена исходная семантика).
- Каждый RPC сверялся 1:1 со старой браузерной логикой и БД перед деплоем.

## 13. AI

**Статус раздела: ✅ Готово (аналитика на эвристиках) · ❌ (LLM/чат)**

### Что умеет (✅)

**AI-аналитика** — детерминированные инсайты, рассчитанные из реальных данных по правилам:
- В **отчётах** (вкладка AI): топ-тариф по продажам, пиковый час, клиенты под угрозой оттока, основной способ оплаты и его доля, истекающие абонементы, средняя выручка в день.

### Как работает

- Инсайты собираются на клиенте из уже посчитанных серверных агрегатов (`reports_*`). Никаких внешних вызовов.

### Модели / ограничения

| Аспект | Факт |
|---|---|
| LLM (Anthropic/OpenAI/др.) | ❌ не подключён (нет SDK в зависимостях) |
| Разговорный AI-чат («Спросите ИИ…») | ❌ **UI-заглушка**: поле `AiChat` очищается по Enter, бэкенда нет; компонент нигде не смонтирован |
| Маршрут `/ai` | ↪ redirect на `/reports` |
| Прогнозы/рекомендации | 📋 заявлено в UI («Скоро»), не реализовано |

> **Вывод:** «AI» в текущей версии — это **аналитика на правилах**, а не искусственный интеллект/LLM. Это важно не выдавать за большее.

## 18. Известные проблемы

**Статус раздела: ✅ Готово (перечень актуален)**

### Открытые задачи / технический долг

- 🚧 **classes/class_bookings** — схема есть, полноценный UI бронирований не завершён.
- 🚧 **База знаний** — каркас без наполнения/редактора.
- 🚧 **Audit-трейл** — пишется/показывается частично.
- ❌ **Разговорный AI-чат** — нефункциональная UI-заглушка.
- ❌ **Роли Reception/Viewer** — заявлены в ТЗ, в системе отсутствуют (есть 6 ролей).
- 🚧 **Мониторинг/аналитика Platform** — частично витрины.

### Ограничения

- **Платежи** — только ручной учёт, без онлайн-шлюзов.
- **SMS** — OTP только через Telegram.
- **Cron** — из-за Vercel Hobby только «раз в сутки» (рассылки/отчёты).
- **PostgREST лимит 1000 строк** — исторический источник багов недоучёта (устранён в отчётах и Platform через RPC; при новых «сырых» выборках учитывать `fetchAllRows`/RPC).
- **Экспорт больших клубов** — грузит сырьё по требованию (~17–50 c на 21k клиентов).

### Риски

- 🔴 **PII без шифрования** — регуляторный/репутационный риск.
- 🔴 **Нет своей стратегии бэкапов** — зависимость от Supabase.
- 🟠 **Нет мониторинга ошибок** — деградации сложно ловить.
- 🟠 **Часть работы не закоммичена в git** (Platform/пагинация/отчёты v0.9 — в рабочем дереве и проде).

## 20. Итоговая оценка проекта

**Статус раздела: ✅ Готово**

### Что уже реализовано

Функциональный **мультитенантный SaaS** с полным ядром CRM (клиенты, абонементы, подписки, посещения, оплаты-учёт, расписание, персонал, склад), гибким RBAC, серверной аналитикой (9 вкладок отчётов), импортом/экспортом, Telegram-ботом и рассылками, а также **полноценной Platform-админкой** (health-score, support-режим, биллинг-заявки). Производительность серьёзно оптимизирована (server pagination, SQL-агрегация, RPC-консолидация).

### Что ещё предстоит

Онлайн-приём платежей, шифрование PII, бэкапы, мониторинг, enforcement тарифов, доведение classes/bookings и базы знаний, а также «настоящий» AI (LLM). Реальный SMS-провайдер, real-time, публичный API, white-label — на следующих фазах.

### Уровень зрелости

**Зрелый MVP на пороге Production.** Продукт демонстрируем и работоспособен на реальных объёмах (проверено на клубе с 21k+ клиентов), но перед платящими клиентами требуется закрыть блокеры Production (платежи, безопасность, бэкапы).

### Оценки по направлениям
