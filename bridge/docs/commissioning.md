# Ввод FitCRM Bridge в эксплуатацию

Эта проверка выполняется на объекте до включения автоматического допуска.

## 1. Подготовка

- Запишите точные модель и firmware контроллера, название и версию серверного ПО.
- Включите официальный API и отдельного пользователя только для чтения событий.
- Синхронизируйте время сервера СКУД, компьютера Bridge и контроллеров по NTP.
- Зафиксируйте ID каждого считывателя и его направление: вход или выход.
- Сделайте резервную копию конфигурации СКУД.

## 2. Проверка Bridge

1. Выполните `npm run doctor -- ./config.json`.
2. Запустите Bridge и откройте `http://127.0.0.1:8787/health`.
3. Убедитесь, что `ok=true`, `queueDepth=0`, а адаптер имеет состояние `running`.
4. Привяжите отдельную тестовую карту к тестовому клиенту в FitCRM.
5. Сделайте один вход и один выход. Сверьте время, направление, клиента и отсутствие дублей.
6. Отключите интернет, сделайте проход, включите интернет и проверьте доставку события из очереди.
7. Повторите один и тот же event ID: второе событие не должно списывать визит.
8. Перезапустите компьютер и убедитесь, что служба стартовала автоматически.

## 3. Допуск и реле

Синхронизация посещений и управление реле — разные функции. По умолчанию Bridge только читает
подтверждённые события прохода. Не подключайте автоматическое открытие до contract-теста
конкретного протокола и отдельной проверки аварийного выхода, пожарного режима и локального
offline-доступа.

## 4. Приёмка

- 20 последовательных проходов без потерь и дублей;
- корректный вход/выход на каждом считывателе;
- очередь восстанавливается после 10 минут без интернета;
- неизвестная карта не создаёт посещение;
- после ротации Bridge-ключа старый ключ получает HTTP 401;
- конфигурационный файл доступен только администратору и сервисной учётной записи.
