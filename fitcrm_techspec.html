01

Продуктовая стратегия

На чём строим систему и почему

Выбранный вектор: Доступная CRM для малых и средних фитнес-залов Узбекистана + AI-удержание клиентов как главный USP против конкурента Deepen.uz. 80% рынка (малые залы) не охвачены Deepen из-за высокого установочного взноса (2–5 млн сум). Мы входим без установочного взноса, с онбордингом за 15 минут.
Целевой рынок
Малые и средние залы
20–200 активных клиентов. Работают в Excel или тетради. Не могут позволить Deepen из-за высокого порога входа. Ташкент — Узбекистан.
Ценообразование
Без установочного взноса
Starter 150–250K сум/мес · Standard 350–500K · Business 700K–1.2M. Deepen берут 2–5M сум единоразово. Это наш главный аргумент при продаже.
Главный USP
AI-удержание + Telegram-first
Предсказываем отток за 7–14 дней. Telegram-бот как основной канал: напоминания, продление, оплата прямо в боте. Никто в СНГ этого не делает.
Приоритет запуска: Выйти с MVP за 10–12 недель. Первые 30–50 платящих клубов в Ташкенте. Затем V2 с AI-удержанием и расширенной аналитикой. Сетевая функциональность — V3.

Важно — payments ↔ subscriptions: Связь односторонняя: payments.subscription_id → subscriptions. Один абонемент может иметь несколько платежей (продление, рассрочка, частичная оплата). Поле payment_id в таблице subscriptions отсутствует намеренно.
Важно — audit_logs.id: Используется bigint вместо uuid — намеренное решение для производительности (INSERT-heavy таблица, последовательный PK быстрее для индексов).
06

Безопасность и шифрование

Обязательные требования для защиты данных клиентов

🔐
Шифрование персональных данных
Поля phone, full_name в таблице clients шифруются на уровне приложения алгоритмом AES-256-GCM до INSERT/UPDATE. В базе хранится только зашифрованный blob. Поиск по телефону — через хеш HMAC-SHA256(phone, pepper) в отдельном индексном поле. Pepper-ключ хранится в Supabase Vault, не в коде.
🗝️
Управление ключами (KMS)
Мастер-ключ шифрования хранится в Supabase Vault (или HashiCorp Vault при self-host). Каждый клуб получает отдельный ключ шифрования данных (DEK). Ключи никогда не пишутся в код или git-репозиторий. Ротация ключей — раз в 90 дней.
🛡️
Row Level Security (RLS)
Supabase RLS включён на всех таблицах. Политика: пользователь видит только данные своего club_id. Прямой доступ к БД без токена — невозможен. Политики проверяются на уровне PostgreSQL, не только в приложении.
🔑
Аутентификация и сессии
JWT-токены с временем жизни access: 15 мин, refresh: 7 дней. Access-токен хранится в памяти приложения (memory-only, не в localStorage — защита от XSS). Refresh-токены хранятся в httpOnly cookie. При смене пароля — инвалидация всех сессий. Поддержка 2FA для роли Owner (TOTP).
🔒
Транспортное шифрование
Все соединения только через TLS 1.3. HTTP — принудительный редирект на HTTPS. API доступен только по HTTPS. Telegram-вебхук — только через HTTPS. Supabase SSL-соединение обязательно (sslmode=require).
🚫
Rate Limiting и защита API
Rate limiting на уровне Vercel Edge: max 100 req/min на IP для публичных endpoints. Защита от brute-force: блокировка на 15 мин после 5 неудачных попыток входа. QR check-in endpoint: max 10 req/min на IP. CSRF-защита на все мутации. Валидация входных данных через Zod.
📋
Audit Log
Каждое изменение данных пишется в audit_logs: кто, что, когда, с какого IP. Логи доступны только Owner. Записи нельзя удалить (только Owner может экспортировать). Хранение — минимум 12 месяцев.
💾
Резервное копирование
Supabase автоматически делает ежедневный бэкап базы данных (Point-in-Time Recovery). Хранение бэкапов — 7 дней (Supabase Pro). Дополнительно: еженедельный pg_dump в зашифрованный S3-bucket (пароль — отдельный от KMS).

Поля с обязательным шифрованием (AES-256-GCM)

-- Таблица: clients full_name → encrypted_full_name bytea + name_hash text (HMAC-SHA256 с pepper для поиска) phone → encrypted_phone bytea + phone_hash text (HMAC-SHA256 с pepper для поиска) -- Таблица: staff salary_info → encrypted_salary bytea (ставки, % — видит только Owner) -- Таблица: payments external_tx_id → encrypted_tx_id bytea (ID транзакции в Click/Payme) -- Таблица: clubs telegram_bot_token → encrypted_tg_token bytea (токен бота — критически важно) -- Ключи шифрования: -- Никогда не хранятся в таблицах БД -- Хранятся в: Supabase Vault → env переменные → только в памяти приложения -- DEK (Data Encryption Key) уникален для каждого club_id -- Pepper для HMAC-SHA256 хранится отдельно от DEK в Vault -- ВНИМАНИЕ: SHA-256 без pepper уязвим для rainbow-таблиц (10^8 UZ телефонов) -- Использовать ТОЛЬКО HMAC-SHA256(value, pepper)
Важно для разработчиков: Никогда не логировать значения зашифрованных полей. В error-сообщениях не выводить персональные данные. Swagger/API-документация не должна содержать примеры с реальными данными. Тестовые данные — только синтетические.

TypeScript-first bot framework, webhook mode. Деплой на Railway / Fly.io (не Vercel — timeout)

QR-коды

Deploy — Frontend
Vercel

Deploy — DB
Supabase Cloud
Регион: Frankfurt или Singapore (ближайший к UZ)
Мониторинг

Ошибки, производительность, Core Web Vitals
CI/CD
GitHub Actions
Тесты → lint → preview deploy → production
Почему этот стек для AI-разработки: tRPC устраняет рассинхронизацию фронта и бэка — Claude Code видит единую типизацию. Prisma schema = единый источник истины о структуре БД. Zod-схемы используются и для валидации, и для генерации типов. Монорепо = один контекст для AI-ассистента.
Критично — Telegram Bot деплой: Grammy.js в webhook mode не совместим с Vercel serverless (25 сек timeout). Бот должен деплоиться на отдельный сервис: Railway, Fly.io или Supabase Edge Functions (для простых обработчиков).
