## v0.9 — Серверная агрегация отчётов + Platform-фиксы (июль 2026)

**Статус: ✅ Готово · задеплоено**

Крупнейшая оптимизация: раздел «Отчёты» полностью переведён с браузерной обработки на серверные SQL-агрегаты (RPC), поэтапно, со сверкой каждой цифры перед деплоем.

**Миграции:** `0028_reports_finance` … `0035_reports_alerts`, `0036_platform_clubs_metrics`.

- ✅ **9 вкладок отчётов → RPC:** финансы, продажи, посещения (heatmap, TZ Asia/Tashkent), клиенты, продления, долги, персонал, «Внимание», AI-инсайты (собираются из готовых агрегатов).
- ✅ **Ленивая загрузка** каждой вкладки + серверный кеш 120 c (`unstable_cache`).
- ✅ **Убрана загрузка сырого массива** на каждый заход в отчёты; экспорт (CSV/PDF) грузит сырьё **по требованию** со статусом «Готовим…».
- ✅ **Результат:** страница отчётов на клубе с 21 199 клиентами — **с ~72 c до 2.2 c**.
- 🐛 **Фикс:** «слепое пятно года» в старом коде (грузил только `created_at ≥ год`) — RPC считает верно (финансы `prevRevenue`, долги старше года).
- 🐛 **Фикс Platform Admin:** в списке клубов «Клиентов» показывалось 0 у крупных клубов — старый код тянул строки `clients`/`visits` без пагинации и упирался в лимит PostgREST 1000 строк. Заменено на RPC `platform_clubs_metrics` (агрегаты count/max). QA-клуб: было 0 → стало 21 199.
- 🐛 **Фикс тёмной темы дашборда:** три компонента (`DashboardTiles`, `RevenueChart`, `DashboardVisitRadial`) содержали захардкоженные светлые цвета → переведены на токены `var(--…)`.
- 🐛 **Фикс экспорта CSV:** `downloadCSV` использовал отсоединённый `<a>` — якорь добавлен в DOM (надёжное скачивание во всех браузерах).
- ⚙️ **Инфраструктура:** `scripts/apply-migration.mjs` — применение миграций через Supabase Management API (без ручной вставки SQL).
